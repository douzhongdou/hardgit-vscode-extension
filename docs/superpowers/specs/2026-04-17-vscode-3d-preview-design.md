# VS Code 三维预览插件设计文档

## 1. 文档信息

- 日期：2026-04-17
- 主题：VS Code 三维预览插件
- 目标：将 `hardgit` 中已验证可用的三维预览能力迁移为一个符合 VS Code 使用习惯的自定义编辑器插件

## 2. 产品定义

本插件是一个运行在 VS Code 编辑区内的三维文件预览器。用户在资源管理器中单击支持的三维文件时，VS Code 使用本插件提供的自定义编辑器，在 preview tab 中直接打开模型预览。

插件的核心定位是“轻量预览器”，不是完整 CAD 工作台。它的目标不是替代专业建模软件，而是在编码、调试、查看仓库资产时，让用户在 VS Code 内快速完成“打开文件并看见模型”的动作。

本项目不是重新发明一套三维渲染系统，而是把 `hardgit` 中已经完成并验证过的三维渲染能力产品化为 VS Code 插件。

## 3. 产品目标

### 3.1 核心目标

- 在 VS Code 中提供稳定、直接、低学习成本的三维模型预览体验
- 复用 `hardgit` 中现有的三维渲染、相机控制、实体树与显示模式能力
- 让用户通过单击文件即可进入预览，而不是跳转外部网站或独立工具
- 在文件更新后自动刷新预览，尽量保持编辑器内工作流连续

### 3.2 成功标准

- 用户单击支持的三维文件后，能在 preview tab 中直接看到模型
- `glb/gltf/step/stp` 均作为正式支持格式可正常显示
- 基础交互顺畅，包括旋转、缩放、平移、适应视图和显示模式切换
- 文件内容变化后默认自动刷新，失败时能给出明确的手动刷新提示
- 插件 UI 与 `hardgit` 保持一致的视觉和操作心智

## 4. 目标用户与使用场景

### 4.1 目标用户

- 处理 `glb/gltf` 资源的前端或工具链开发者
- 处理 `step/stp` 图纸或模型文件的工程类开发者
- 希望在代码仓库中就地查看三维文件的团队成员

### 4.2 典型场景

1. 用户在 VS Code 的 Explorer 中单击一个三维文件，希望立刻在编辑区看到它。
2. 用户希望在不离开 VS Code 的前提下检查模型是否正确、方向是否正常、轮廓是否合理。
3. 用户在构建流程或脚本重新生成模型文件后，希望当前预览自动更新。
4. 用户希望通过实体树快速定位模型结构并进行选中联动。

## 5. 支持格式

以下格式在第一版中作为正式支持格式：

- `glb`
- `gltf`
- `step`
- `stp`

格式策略的前提是：`hardgit` 已经完成并验证过这套三维渲染链路，只要图纸本身没有问题，即可正确显示。因此，格式支持在产品定义中不标记为实验性能力。

## 6. 打开方式与编辑器行为

### 6.1 打开方式

- 用户在 Explorer 中单击支持格式文件
- VS Code 使用本插件的自定义编辑器打开该文件
- 打开位置为编辑区 preview tab

这里的“单击预览”本质上依赖 VS Code 的编辑器预览机制，而不是额外设计一个单独的浮层预览入口。

### 6.2 与 VS Code 二进制文件行为的关系

对于二进制文件，VS Code 默认文本编辑器可能无法直接显示内容。但如果插件为相应文件类型注册了自定义编辑器，则该编辑器可以作为这些文件的默认编辑器或可选编辑器。

这意味着：

- 本插件可以通过 `customEditors` 贡献点声明支持的文件类型
- 本插件可以作为这些文件的专用查看编辑器
- 用户也可以通过 VS Code 的“Reopen With”或“选择编辑器”能力切换到本插件

### 6.3 编辑器复用策略

- 同一个文件重复单击时，优先复用当前 preview editor
- 用户双击标签页后，该标签页转为固定标签页，行为与 VS Code 原生编辑器一致

## 7. MVP 功能范围

### 7.1 第一版保留功能

- 自定义编辑器打开三维文件
- 模型渲染主画布
- 鼠标旋转、缩放、平移
- 自动适应视图
- 手动适应视图
- 重置视角
- 背景切换
- 显示模式切换：`solid / wireframe / edges`
- 实体树展示
- 实体树与场景选中联动
- 加载中状态
- 加载失败状态
- 文件变化后的自动刷新
- 自动刷新失败时的手动刷新提示

### 7.2 第一版明确不做

- 批注
- 剖切
- 云端能力
- 登录能力
- 上传本地文件入口
- URL 加载入口
- 多人协作
- 模型编辑回写
- 截图导出
- 文件对比

## 8. UI 设计原则

### 8.1 总体原则

编辑器 UI 基本沿用 `hardgit` 现有视觉和布局，不重新设计一套新的产品界面。

目标不是把 `hardgit` 改造成“更像插件的新页面”，而是在迁移到 VS Code 后，尽量保持用户已经认可的界面长相和交互心智。

### 8.2 保留内容

- 主画布布局
- 顶部工具栏整体风格
- 背景与场景观感
- 显示模式切换方式
- 实体树面板样式与使用方式

### 8.3 收口原则

虽然 UI 视觉沿用 `hardgit`，但首版仍然需要按“轻量预览器”定位裁掉不属于该定位的能力：

- 不保留批注面板
- 不保留剖切面板
- 不保留上传文件和 URL 加载入口
- 不保留与云端或站点业务相关的界面元素

## 9. 用户交互流程

### 9.1 主流程

1. 用户在 Explorer 中单击 `glb/gltf/step/stp` 文件。
2. VS Code 调用本插件的自定义编辑器，在 preview tab 中打开文件。
3. 扩展端获取当前文档 URI，并将可供 Webview 访问的资源信息发送给 Webview。
4. Webview 中的编辑器应用初始化三维预览内核。
5. 预览内核按文件扩展名选择对应加载器。
6. 画面先显示加载状态。
7. 模型加载成功后自动执行一次适应视图。
8. 用户开始进行旋转、缩放、平移、背景切换和显示模式切换。
9. 如果启用了实体树，模型加载完成后自动构建实体树并支持联动选中。
10. 如果文件后续发生变化，编辑器默认自动刷新；若刷新失败，则提示用户手动刷新。

### 9.2 关键交互原则

- 首次打开后自动适应视图
- 加载失败时必须显示明确错误态，不能白屏
- 文件更新优先自动刷新，不增加用户负担
- 自动刷新失败时提供明确的恢复路径

## 10. 模块保留、裁剪与重构

### 10.1 保留模块

以下内容优先从 `hardgit` 中保留或直接迁移：

- `MaLiang` 的整体画布组织方式
- `ModelLoaderR3F`
- 相机控制与 `fitView` 逻辑
- `solid / wireframe / edges` 渲染模式切换
- 实体树及选中联动
- 当前已经验证可用且用户满意的 UI 布局与样式

### 10.2 裁剪模块

以下内容不进入第一版：

- 批注相关模块
- 剖切相关模块
- Cloud / 登录 / 文件服务相关逻辑
- 上传文件与 URL 加载入口
- 与站点页面、业务路由或服务端状态强绑定的逻辑

### 10.3 重构适配点

从 `hardgit` 迁移到 VS Code 插件时，需要完成以下适配：

- 文件输入来源从“页面上传或 URL”改为“VS Code 当前文档 URI”
- 刷新机制从页面内控制改为文档变更驱动
- 错误提示从网页日志思维改为编辑器可见的状态提示
- 资源路径适配 Webview URI 和 CSP 约束
- 主题适配尽量兼容 VS Code 浅色与深色主题

## 11. 系统结构设计

建议整体结构拆为三层：

### 11.1 Extension 层

职责：

- 注册 VS Code `Custom Editor`
- 处理编辑器生命周期
- 监听工作区中文档变更
- 将文件 URI、资源路径和刷新消息发送给 Webview

### 11.2 Editor App 层

职责：

- 运行在 Webview 中
- 负责整体编辑器页面壳
- 保持与 `hardgit` 一致的 UI 呈现
- 接收扩展端消息并驱动预览内核

### 11.3 Viewer Core 层

职责：

- 管理三维场景
- 加载模型
- 控制相机
- 控制显示模式
- 维护实体树与场景联动

设计原则：

- `viewer-core` 不依赖 Next.js
- `viewer-core` 不依赖站点路由
- `viewer-core` 不依赖后端服务
- `viewer-core` 只保留纯预览能力

## 12. 自动刷新策略

刷新策略明确为：

- 文档发生变化后默认自动刷新
- 自动刷新失败时提示用户手动刷新

这样做的原因是：

- 用户对预览器的预期是“文件更新后内容同步变化”
- 在构建生成模型文件的场景里，自动刷新能显著减少切换成本
- 失败时保留手动刷新入口，避免状态卡死

## 13. 风险与关注点

### 13.1 主要风险

- VS Code Webview 的资源访问、URI 映射与 CSP 约束
- 大模型文件的加载体验与主线程卡顿风险
- `hardgit` 现有代码中与站点逻辑的耦合
- 自定义编辑器生命周期与状态同步细节

### 13.2 风险应对

- 尽早将资源路径设计为 Webview 可控映射
- 将模型加载、错误态和刷新态纳入标准 UI 流程
- 迁移时坚持“保留预览能力，裁掉站点能力”
- 先打通自定义编辑器生命周期，再逐步补全功能细节

## 14. 分阶段实施建议

### Phase 1：打通 VS Code 打开链路

- 注册自定义编辑器
- 支持单击文件后在 preview tab 中打开
- 接入 `glb/gltf/step/stp` 文件
- 能显示模型、加载态和错误态

### Phase 2：迁移主要预览能力

- 接入 `hardgit` 现有 UI 布局
- 接入工具栏
- 接入相机控制
- 接入显示模式切换
- 接入实体树与联动选中

### Phase 3：补齐编辑器体验

- 接入文件变化自动刷新
- 增加刷新失败后的手动恢复提示
- 完成 Webview 资源映射和主题细节适配
- 收敛整体预览体验

## 15. 结论

本项目的最佳方向不是重新做一个三维站点，也不是实现完整 CAD 工作台，而是把 `hardgit` 中已经成熟的三维预览能力封装成一个符合 VS Code 使用习惯的自定义编辑器。

第一版产品边界已经明确：

- 打开方式是单击文件后在 preview tab 中打开
- UI 基本沿用 `hardgit`
- 正式支持 `glb/gltf/step/stp`
- 保留实体树
- 不做批注
- 不做剖切
- 文件变化后自动刷新，失败时支持手动刷新

在这个边界下，项目既能最大化复用现有成果，也能保持作为 VS Code 插件应有的清晰定位。
