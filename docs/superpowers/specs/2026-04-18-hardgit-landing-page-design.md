# Hardgit 插件介绍页设计文档

## 1. 背景

当前项目已经具备可打包的 VS Code 3D 模型预览插件能力，但还缺少一个面向外部用户的静态介绍页。这个页面将部署到 GitHub Pages，用于让首次接触项目的用户快速理解插件用途，并通过 GitHub Releases 下载试用版本。

本页面不是开发文档站，也不是产品后台，而是一个单页式产品介绍落地页。

页面需要同时支持中文与英文展示，并为后续扩展更多语言预留结构。

## 2. 页面目标

本页面只服务一个核心目标：

- 让机械 / 结构 / CAD 工程师在 30 秒内理解插件价值，并点击进入 GitHub Releases 下载试用

围绕这个目标，页面需要完成以下任务：

- 用一句清晰的话说明插件解决什么问题
- 用一张真实截图建立可信度
- 用少量高价值功能点说明插件能力
- 用工程场景语言说明适用人群和使用方式
- 在首屏和页尾都提供明确下载入口
- 让中文与英文用户都能无障碍浏览页面内容

## 3. 目标用户

目标用户为机械、结构、CAD 相关工程师，以及需要在仓库和模型文件之间频繁切换的人。

这些用户通常具备以下特征：

- 已经在使用 VS Code 管理代码、脚本、说明文档或工程资产
- 经常需要查看 `STEP / STP / GLB / GLTF` 文件
- 不希望为了快速确认模型结构而频繁切换到独立 CAD 或 3D 工具
- 更看重效率、清晰度和可用性，而不是炫技型视觉表现

因此页面语言需要偏工程工具表达，不走夸张营销话术。

## 4. 页面定位

页面定位为 GitHub 风格的极简产品落地页。

页面应传达的核心印象：

- 这是一个实用、克制、专业的工程插件
- 重点在“能立刻帮你看模型”
- 使用门槛低，下载路径明确
- 即使切换语言，页面结构和信息密度也保持一致

页面不承担以下职责：

- 不作为完整使用手册
- 不承载复杂更新日志
- 不展示过多技术实现细节
- 不做品牌宣传站式的大量叙事

## 5. 核心信息主张

页面的核心主张为：

> 在 VS Code 中直接预览 STEP、STP、GLB、GLTF 模型，不必频繁切换软件。

辅助信息主张：

- 用只读预览方式查看模型，不改动源文件
- 适合在工程仓库、脚本与模型之间快速切换
- 支持实体树、显示模式切换、背景切换和视图重置

## 6. 页面结构

整个页面采用单页结构，自上而下分为五个主要区块。

### 6.1 Hero 首屏

目标：

- 第一时间说明插件用途
- 用截图快速建立可信度
- 立即提供下载入口

内容：

- 主标题：强调“在 VS Code 里直接预览 3D 模型”
- 副标题：说明支持的格式与典型工程收益
- 主按钮：跳转 GitHub Releases
- 次按钮：跳转 GitHub 仓库
- 右侧展示真实截图：`public/assets/snapshot.png`

首屏要控制信息密度，不堆过多段落。用户进入页面后，应在一屏内看到“是什么、适合谁、去哪下载”。

### 6.2 功能卖点区

目标：

- 用最少的内容说明插件确实有工程使用价值

建议展示 6 个功能点：

- 直接打开 `glb / gltf / step / stp`
- 只读预览，不改源文件
- 实体树浏览与节点可见性切换
- `Solid / Wireframe / Edges` 三种显示模式
- 明暗背景切换
- `Fit View` 与刷新当前预览

表现形式：

- 2 列或 3 列卡片
- 每张卡片只有标题和一行说明
- 不使用复杂图标系统，可使用简洁圆点、线性小图形或纯文本编号

### 6.3 使用场景区

目标：

- 让目标用户把插件和自己的工作流建立联系

建议展示 3 个典型场景：

- 在仓库中快速检查结构件和装配文件
- 审核 `STEP / STP` 模型时快速确认结构与层级
- 在代码、说明文档和模型文件之间连续工作，减少切软件成本

这一部分要用工程师熟悉的话说话，不要写泛泛的“提升效率”“更智能”。

### 6.4 支持格式与使用方式区

目标：

- 清楚说明插件现在能做什么，避免用户误解

内容：

- 支持格式：`glb`、`gltf`、`step`、`stp`
- 安装方式：从 GitHub Releases 下载 `.vsix`
- 使用方式：在 VS Code 中打开对应文件，即可进入自定义预览

这一部分可以加一段简短说明：

- `gltf` 依赖外部资源时，需要纹理与 `.bin` 文件路径完整
- 部分超大 STEP 装配可能存在仅解析结构、无法生成可渲染几何的情况

这能提前设置预期，减少下载后的误判。

### 6.5 页尾 CTA

目标：

- 在用户浏览完页面后，再给一次明确下载入口

内容：

- 一句收束文案
- “前往 GitHub Releases”按钮
- “查看项目仓库”文本链接

### 6.6 语言切换区

目标：

- 让用户能在中文和英文之间直接切换，不需要跳转到不同站点

内容：

- 页面头部提供简洁语言切换入口
- 默认语言优先中文，保留英文切换
- 切换后页面所有核心文案同步更新，包括按钮、标题、功能点、场景区和页尾 CTA

表现形式：

- 使用 GitHub 风格的轻量文本切换器或分段按钮
- 不做复杂国际化组件，不加入国旗图标，不喧宾夺主

## 7. 视觉设计方向

整体视觉参考 GitHub 官网与 GitHub Docs 的极简风格，但不机械复制。

视觉原则：

- 大留白
- 清晰分栏
- 浅色背景
- 深色正文
- 蓝色作为交互强调色
- 细边框、轻阴影或几乎无阴影
- 不使用夸张渐变和装饰性背景

### 7.1 色彩

建议色彩方向：

- 页面背景：接近白色或极浅灰
- 正文标题：深灰接近黑色
- 辅助文字：中灰
- 边框：极浅灰
- 链接与主按钮：GitHub 风格蓝色

整体应保持“文档感”和“产品感”的平衡，避免视觉过于营销化。

### 7.2 字体

优先使用系统字体栈，保持 GitHub 风格的稳定和兼容性：

- 中文：`"PingFang SC"`, `"Microsoft YaHei"`, `sans-serif`
- 英文：系统 UI 字体栈

标题、正文、按钮都保持克制，不做强风格字体实验。

### 7.3 组件风格

组件应尽量简洁：

- 按钮为圆角矩形，主次层级明确
- 卡片使用浅边框和较小圆角
- 截图外框类似 GitHub 的预览容器
- 分割线、间距与段落层级比颜色更重要
- 语言切换器应与导航或首屏操作区自然融合，不额外制造视觉焦点

### 7.4 动效

动效只保留最低限度：

- 按钮 hover
- 卡片轻微 hover
- 页面区块轻微淡入

不引入复杂滚动特效、视差、粒子背景或强动画。

## 8. 内容策略

页面文案必须短、清楚、工程导向。

文案策略：

- 少用抽象形容词
- 多用具体文件格式、具体工作场景、具体操作能力
- 标题强调“直接预览”“在 VS Code 中”“支持 STEP / GLTF / GLB”
- 避免“革命性”“下一代”“极致体验”之类空泛营销语
- 中文与英文版本在信息层级上保持等价，不出现一边完整、一边简化过度的情况

推荐内容顺序：

- 是什么
- 适合谁
- 能做什么
- 怎么下载

多语言策略：

- 第一阶段支持 `zh-CN` 与 `en`
- 文案采用统一键值结构管理，避免中英文写成两份完全独立页面
- 页面切换语言时不刷新整页，只更新文案内容
- 后续若扩展更多语言，应只新增语言资源，不重写页面结构

## 9. 技术方案

页面采用纯静态实现，适配 GitHub Pages。

建议文件结构：

- `site/index.html`
- `site/styles.css`
- `site/main.js`
- `site/i18n.js` 或 `site/content.js`
- `site/assets/snapshot.png`

实现原则：

- 不依赖前端框架
- 不引入打包工具
- 使用相对路径资源
- 可直接通过 GitHub Pages 部署
- 语言文案集中管理，避免散落在 HTML 各处造成后续维护困难

如果后续需要自动化部署，可在仓库中增加 GitHub Actions，将 `site/` 目录发布到 Pages。

语言实现建议：

- HTML 结构中为主要文本节点保留稳定的 `data-i18n` 标识
- `main.js` 负责读取当前语言、渲染文案并绑定语言切换逻辑
- 可使用 URL 查询参数、hash 或本地存储记住语言选择
- 若无用户偏好，则默认显示中文

## 10. 资源与素材

当前确认使用的素材：

- 页面主截图：`public/assets/snapshot.png`

处理方式：

- 构建介绍页时复制或引用该截图到页面资源目录
- 保持截图为页面主视觉，不再额外要求多张截图才能上线

## 11. 非目标

本次页面不包含以下内容：

- 博客系统
- 评论系统
- 在线演示
- 复杂安装引导向导
- 自动检测用户平台并下载对应资源
- 丰富图库或多截图轮播

说明：

- 页面需要支持中英双语与后续扩展多语言，但不在本次引入复杂国际化框架

这些都不是当前页面完成“快速理解并下载试用”所必需的功能。

## 12. 验收标准

页面完成后，应满足以下标准：

- 用户进入首屏后 10 秒内能知道插件用途
- 页面核心下载入口明确，并指向 GitHub Releases
- 页面整体风格简洁，接近 GitHub 极简风
- 页面能够在桌面与移动端正常浏览
- 截图展示清晰，不被过度裁切
- 页面文案面向机械 / 结构 / CAD 工程师，而不是泛用开发者营销话术
- 页面可直接部署为 GitHub Pages 静态站点
- 页面支持中文与英文切换，且两种语言的内容完整度一致
- 多语言结构可继续扩展，而不需要重做整页 HTML

## 13. 后续实现建议

后续实现阶段应优先关注以下事项：

- 先完成静态单页，再考虑自动部署
- 优先把首屏和 CTA 做扎实，不先追求复杂视觉
- 保持资源路径和 GitHub Pages 部署路径兼容
- 下载按钮、仓库链接、截图资源都做成明确、可替换的常量位
- 中英文文案管理先做干净，再考虑加入第三种语言

这会让后续升级页面内容和切换发布地址更轻松。
